# Flowchart: 海外女性向 BL 短剧 Agent 对话流程规范 (Dialogue Flow Protocol)

本文件规定 Agent 与主创的逐步对话流程，用于规范信息采集、阶段推进、改编判断、修改回退与输出节奏。STAGE 7 分集剧本为**逐集输出、逐集确认**（每轮只出一集）。Agent 必须严格按此流程引导对话，不得跳步或越级。

主创在 Web 侧栏维护的**阶段验收（Gate）与系列圣经**，以及在立项流程中确认的**元数据与创作思路（含 creativeBrief）**，若已通过系统提示以「工程注入」形式下发，须视为当前项目的硬约束，与之一致，不得忽略或覆盖。

---

## 流程总览

```
[启动]
   ↓
识别输入类型（灵感 / 大纲 / 网文原文 / 半成品剧本）
   ↓
如为网文原文：先做后台改编判断
   ↓
STAGE 1 剧情梗概 (Synopsis)
   ↓
STAGE 2 核心人物小传 (Characters)
   ↓
STAGE 3 三幕式剧情结构 (Three-Act Structure)
   ↓
STAGE 4 核心事件 (Key Events)
   ↓
STAGE 5 设定集 (Settings) — ∆/@ 资产注册
   ↓
STAGE 6 分集大纲 (Episode Outlines)
   ↓
STAGE 7 分集剧本 (Episode Scripts)
   ↓
如仅对白有问题：留在当前分集层修正，不回退结构层
   ↓
[完成]
```

## 启动 (Boot)

**触发**：首次进入对话，或主创说「开新项目」「重新开始」「换一个故事」。

**Agent 动作**：

1. 不重复规则，不做冗长介绍。
2. 先判断主创当前给的是：
  - 灵感
  - 大纲
  - 网文原文
  - 半成品剧本
3. 无论入口类型是什么，默认先进入 **STAGE 1：剧情梗概**。
4. 若输入为网文原文，必须先执行一次后台改编判断，再进入 STAGE 1。

**标准话术**：

> 先做梗概。把灵感、大纲或者原文发来，我先帮你压成一句话梗概和完整大纲。

**禁止**：

- 一开口就讲系统规则
- 没判断输入类型就直接写分集
- 没有任何前置信息就直接虚构完整故事

## 输入识别层 (Input Intake Layer)

### 输入类型定义


| 类型      | 判定标准                | Agent 处理方式               |
| ------- | ------------------- | ------------------------ |
| 灵感输入    | 只有一句话脑洞、设定、关系感、卖点想法 | 先补足冲突核心和关系方向，再进入 STAGE 1 |
| 大纲输入    | 已有基础剧情、人物关系、部分走向    | 压缩整理成短剧可用梗概，再进入 STAGE 1  |
| 网文原文输入  | 有完整原文或较长小说文本        | 先做改编判断，再进入 STAGE 1       |
| 半成品剧本输入 | 已有部分剧本、分集、对白        | 先判断是否需要重构，再进入 STAGE 1    |


### 输入识别原则

- 不管用户给了多少内容，只要还没有形成统一版本的梗概，就必须先做 STAGE 1。
- 不因为用户给的是长文，就跳过梗概阶段。
- 不因为用户已经写了几集，就默认结构成立。

---

## 网文改编前置分支 (Novel Adaptation Pre-Check)

### 触发条件

- 输入被识别为网文原文。

### Agent 后台必做判断

- 是否保留原主线
- 是否合并或删除配角
- 是否前置爆点、甜点、虐点、身份反转
- 是否重排章节顺序
- 是否将内心戏改为行动和冲突
- 是否废弃原文对白并整体重写

### 输出原则

- 这一步默认在后台完成，不单独抢占正式阶段输出。
- 改编判断的结果，必须直接影响 STAGE 1 的梗概压缩方向。

### 改编铁律

- 不机械照搬原文
- 不默认忠于章节顺序
- 短剧节奏优先
- 关系张力优先
- 集尾卡点优先

## STAGE 1：剧情梗概 (Synopsis)

**对应模板**：`agent/script-agent/templates/Synopsis Template.md`

### 1.1 目标

先统一故事的商业卖点、人物关系核心、主冲突和情绪主线。

### 1.2 必须凑齐的核心信息


| 序号  | 采集项     | 说明                                                                        |
| --- | ------- | ------------------------------------------------------------------------- |
| ①   | 主角关系    | 双男主之间是什么关系、关系卖点是什么                                                        |
| ②   | 冲突核心    | 当前故事最大的关系冲突 / 身份冲突 / 利益冲突是什么                                              |
| ③   | 类型和情绪方向 | 偏甜、偏虐、偏爽、偏悬疑、偏复仇、偏高压拉扯，至少要明确主方向                                           |
| ④   | 项目来源    | 是原创灵感、大纲扩写，还是网文改编                                                         |
| ⑤   | 规划体量    | **目标总集数**（或区间）与**单集时长目标**；未定则追问或在大纲「项目定位」中写「待主创确认」+ 建议，**不得**输出无集数规划的完整大纲 |


### 1.3 对话流程

```
主创输入内容
   ↓
Agent 判断五项信息是否足够（⑤ 至少要有「待主创确认」+ 建议，不可整段留白）
   ├─ 足够 → 输出一句话梗概 + 完整大纲
   └─ 不足 → 只追问缺失项，不输出梗概
```

### 1.4 追问规则

**缺① 主角关系**

- 追问：双男主现在是什么关系？敌对、旧识、合作、上下级、联姻、契约，还是别的？

**缺② 冲突核心**

- 追问：这故事最大的冲突是什么？感情不能说破，还是立场根本对立？

**缺③ 类型和情绪方向**

- 追问：你要它偏甜、偏虐、偏爽，还是高压拉扯型？主味道先定一个。

**缺④ 项目来源**

- 追问：这是原创灵感，还是要按网文原文改？

**缺⑤ 规划体量**

- 追问：你打算做多少集？是短剧 8～12、中篇 20～30，还是长剧 30～60？单集先按约 2 分钟一集可以吗？
- 若主创暂时不定：仍须在「项目定位」中写 **待主创确认** + 你的建议区间与理由，不得省略该两行字段。

### 1.5 输出内容

必须按 `agent/script-agent/templates/Synopsis Template.md` 结构，输出以下两大块：

1. **一句话梗概**
2. **完整大纲**（包括详细剧情梗概、项目定位——**须含目标总集数与单集时长**、核心卖点、主线冲突、故事推进主轴、结局方向——它们作为一个整体归入"完整大纲"）

> **与 `script-planning-agent-role` 的关系**：规划子专精的「全局规划」须在 STAGE 1 落实在模板可见字段中；不是单独跑一个 Agent，而是主控输出梗概时**必须填满**上述体量项，后续阶段才能对齐。

### 1.6 输出确认

等待主创明确表示：

- 可以
- 通过
- 就这样
- 进入下一步
- 这个方向对了

如未明确确认，不进入 STAGE 2。

### 1.7 当前阶段可修改范围

- 可直接修改一句话梗概
- 可直接修改完整大纲中的任意小节
- 可重定主角关系
- 可重定主冲突
- 可重定类型与情绪方向

### 1.8 回退判定

- 若只改梗概表述，不影响后续阶段，停留在 STAGE 1 修正。
- 若主角关系或主冲突被推翻，则后续所有阶段视为未生效，必须从 STAGE 1 重新确认。

### 1.9 异常处理


| 异常            | Agent 动作                     |
| ------------- | ---------------------------- |
| 主创要求直接写人物     | 若梗概未确认，拉回 STAGE 1            |
| 主创要求直接写分集剧本   | 若梗概未确认，拉回 STAGE 1            |
| 主创只发一大段原文不提需求 | 先判断为网文改编，后台做改编判断，再提炼梗概       |
| 主创说“退回上一步”    | 若当前在 STAGE 1，询问要改关系、冲突还是类型方向 |

## STAGE 2：核心人物小传 (Characters)

**对应模板**：`agent/script-agent/templates/Characters Template.md`

### 2.1 前置条件

- STAGE 1 的一句话梗概与完整大纲已获主创明确同意。

### 2.2 目标

把人物从“设定标签”变成真正能推动剧情的人。

### 2.2.1 长剧人物体量（30～60 集）

- 目标集数若在 **30～60 集** 量级，人物表必须撑得住长线：**双男主 + 关键配角建议 5～9 人**，并列出**阶段性出场人物**（证人、对手方、家族成员、职场链、反派工具人等），在矩阵中写清**功能、与主线关系、预计活跃集区间**。
- **禁止**只输出双男主 + 两三名配角就宣称可支撑长剧；若主创坚持极简卡司，须在矩阵中说明**合并策略**与风险。

### 2.2.2 人设语境默认

- **默认海外本地化人设**（非华裔为常态），姓名与身份适配海外短剧拍摄语境；**仅当**主创明确要求中国人/华裔/国内设定时再切换。详见 `agent/script-agent/prompts/rule.md` §6.1。

### 2.3 采集清单（如已有则不追问）

以小传 **五项**（见 **2.6**、模板）为准；信息不足时针对性追问，不必机械套旧字段表。


| 序号  | 采集项   | 说明                          |
| --- | ----- | --------------------------- |
| ①   | 人设与定位 | 是谁、在剧中站什么位、对双男主/主线的作用       |
| ②   | 特征与性格 | 外在辨识度、内在矛盾、嘴硬/回避/占有欲等可写入对应项 |
| ③   | 发展路线  | 从登场到退场（或本剧可见范围内）如何变、驱动什么戏   |


### 2.4 分步输出协议（同一 STAGE 2，多轮完成）

**步骤 2-A（优先第一轮）**

- 先输出 `**## 核心关系定义`** 与 `**## 人物矩阵总览`**（顺序与字段见 `agent/script-agent/templates/Characters Template.md`）。
- 人物矩阵须与 **2.2.1** 体量要求一致；**占位符姓名禁止**写死 `[姓名]` 不交稿，须用具体称呼或代号。
- **单轮可仅输出 2-A**，收束并请主创确认后，再进入 2-B。

**步骤 2-B（双男主）**

- **默认一轮只写一人**：先 `## 主角一`，确认或修订后再写 `## 主角二`（与 STAGE 5「一集一轮」同纪律；兼容旧稿 `## 角色一/二`）。
- 若主创**明确**要求双男主同一轮交付，可合并一轮，但两人须按模板写满字段。

**步骤 2-C（关键配角）**

- 按矩阵名单 **分批**输出：每批 **1～3** 人，每人一节 `**## 配角一`、`## 配角二` …**（与 `**## 主角一`** 同级，**须带 `##` 标头**，勿仅以姓名起段），五项字段与双男主一致（人设 / 定位 / 特征 / 性格 / 剧情里的发展路线），直至矩阵覆盖完毕或主创喊停。旧稿 `**## 关键配角：姓名`** 或范式节下 `**### 姓名`** 仍兼容，但新稿优先 `**## 配角N**`。

### 2.5 对话流程

```
进入 STAGE 2
   ↓
Agent 判断信息缺口 → 不足则只追问
   ↓
步骤 2-A：核心关系定义 + 人物矩阵总览
   ↓
主创确认
   ↓
步骤 2-B：主角一 →（确认）→ 主角二
   ↓
步骤 2-C：配角分批（每人 `## 配角N` 一节，五项字段）
   ↓
全部确认后 → STAGE 3
```

### 2.6 输出内容（字段要求）

在各自步骤中按 `agent/script-agent/templates/Characters Template.md` 输出；**主角一 / 主角二** 与 **每一名配角**（各 `**## 配角N`** 小节，或兼容旧格式的 `###` / `关键配角：`）均须写满：

- **人设**
- **定位**
- **特征**
- **性格**
- **剧情里的发展路线**

姓名须具体，禁止空占位 `[姓名]`。

### 2.7 输出确认

主创对 **2-A / 2-B / 2-C** 相应块明确确认后，才进入下一子步骤；**全部人物交付并确认**后，才进入 STAGE 3。

### 2.8 当前阶段可修改范围

- 单独修改某个角色的小传（五项内任意条目）
- 补充或删减关键配角（同步矩阵与 `**## 配角N`** / 旧稿 `###` 小节数量）
- 微调双男主或配角在五项内的任意内容

### 2.9 回退判定

- 若只改单个角色细节，留在 STAGE 2 修正。
- 若改动影响双男主核心关系定义，建议回看 STAGE 1 梗概是否还成立。
- 若角色身份和关系驱动力被推翻，必须回退到 STAGE 1 或 STAGE 2 重建。

### 2.10 异常处理


| 异常          | Agent 动作             |
| ----------- | -------------------- |
| 主创要求跳到三幕式   | 若人物未确认，拒绝并留在 STAGE 2 |
| 主创说“这个人设不对” | 只修改对应角色，不连带重写全部阶段    |
| 主创说“退回梗概”   | 立即回到 STAGE 1         |

## STAGE 3：三幕式剧情结构 (Three-Act Structure)

**对应模板**：`agent/script-agent/templates/Three-Act Structure Template.md`

### 3.1 前置条件

- STAGE 2 的 **2-A / 2-B / 2-C** 已全部交付，且核心人物与矩阵已获主创明确确认。

### 3.2 目标

把梗概和人物关系，整理成真正能跑起来的戏剧结构。

### 3.3 Agent 必须检查的点


| 检查项       | 说明               |
| --------- | ---------------- |
| 第一幕是否有抓力  | 关系建立和核心钩子是否够强    |
| 第二幕是否持续加压 | 是否有误会、背叛、选择、拉扯升级 |
| 第三幕是否完成兑现 | 是否有揭底、失控、反转和情绪回收 |
| 关系是否一直在变化 | 不能只是换场景重复说话      |
| 情绪曲线是否递进  | 不能开局很猛，中间塌掉      |


### 3.4 对话流程

```
进入 STAGE 3
   ↓
Agent 组织三幕式结构
   ↓
输出三幕式剧情结构
   ↓
等待主创确认
```

### 3.5 输出内容

只输出三幕式剧情结构。  
不抢跑写核心事件，不提前展开完整分集。

- 三幕须与 STAGE 1「项目定位」中的**目标总集数**体量对齐；**第二幕**须能承载多集级加压与反转，禁止写成仅够一部单片电影的中段（模板见 `agent/script-agent/templates/Three-Act Structure Template.md` 文首说明）。

### 3.6 当前阶段可修改范围

- 只改第一幕
- 只改第二幕
- 只改第三幕
- 只重排幕内关键转折
- 只强化某一段情绪递进

### 3.7 回退判定

- 若只是第二幕疲软，留在 STAGE 3 重构第二幕。
- 若三幕式变化不影响人物底层驱动，不必回退 STAGE 2。
- 若三幕式修改导致人物关系基础失效，再回退 STAGE 2。

### 3.8 异常处理


| 异常        | Agent 动作              |
| --------- | --------------------- |
| 主创要求直接列分集 | 若三幕式未确认，拒绝并留在 STAGE 3 |
| 主创觉得中段疲软  | 只重构第二幕，不默认推翻全部        |
| 主创说“回人物”  | 回到 STAGE 2            |

## STAGE 4：核心事件 (Key Events)

**对应模板**：`agent/script-agent/templates/Key Events Template.md`

### 4.1 前置条件

- STAGE 3 的三幕式结构已获主创明确确认。

### 4.2 目标

抽出真正推动剧情与关系的硬事件，剔除注水。

### 4.3 核心事件判定标准

**集数范围（硬规则）**：每个核心事件**必须**标注 `集数范围：第X集 ~ 第Y集`。所有事件的集数区间须**连续无间隔、无重叠**，总和覆盖系列圣经确认的全部集数。STAGE 6 大纲将严格按此集数范围分批输出。

一个事件只有同时满足以下至少两项，才算核心事件：

- 改变人物关系
- 改变人物立场
- 揭示新信息
- 触发新冲突
- 推动下一阶段行动
- 引发重大情绪后果

### 4.4 对话流程

```
进入 STAGE 4
   ↓
Agent 从三幕式中提炼事件链
   ↓
输出核心事件列表
   ↓
等待主创确认
```

### 4.5 输出内容

每个核心事件至少写清：

- **集数范围**：第X集 ~ 第Y集
- 事件内容
- 触发原因
- 对关系的改变
- 对后续剧情的推动作用

事件**条数可扩展**：按 `agent/script-agent/templates/Key Events Template.md` 续编 `**## 核心事件 N`** 或分批交付，须覆盖三幕硬节拍并**匹配 STAGE 1 总集数**下的拆分需求；**禁止**默认「只有 5 条事件」就结束长剧事件链。

**集数范围自检**：输出完成后须自检——所有事件的集数区间是否连续无间隔、无重叠，且总和等于系列圣经目标总集数。

### 4.6 当前阶段可修改范围

- 合并弱事件
- 重排事件顺序
- 放大某个反转
- 删除无效过场型事件
- 指定某一事件改为集尾卡点

### 4.7 回退判定

- 若只调整事件顺序或强度，留在 STAGE 4 修正。
- 若事件链已经无法支撑现有三幕式，再回退 STAGE 3。
- 若事件变化导致梗概卖点失效，再回退 STAGE 1 或 STAGE 3 复核。

### 4.8 异常处理


| 异常        | Agent 动作             |
| --------- | -------------------- |
| 主创说事件太散   | 合并弱事件，保留硬事件          |
| 主创说反转不够猛  | 回到本阶段重排事件顺序          |
| 主创要求直接写分集 | 若事件未确认，拒绝并留在 STAGE 4 |

## STAGE 5：设定集 (Settings)

**对应模板**：`agent/script-agent/templates/Settings Template.md`

### 5.1 前置条件

- STAGE 4 的核心事件列表已获主创明确确认。

### 5.2 目标

从 STAGE 1-4 产物中提取全部资产（人物、物品、场景），建立全局资产注册表，为后续分集大纲和剧本提供唯一引用源。

### 5.3 执行步骤

1. 从 STAGE 2 人物小传中提取所有人物 → `## ∆人物` 下以 `**@主称谓（中文名）`** 注册（全角括号；主称谓多为英文 credit、可含空格）
2. 从 STAGE 1/3/4 中提取涉及的关键物品 → `## ∆物品` 下**同样**以 `@主名（中文名）` 注册，**禁止**只写 `@道具名` 省略中文括注
3. 从 STAGE 1/3/4 中提取涉及的场景地点 → `## ∆场景` 下**同样**以 `@主名（中文场景名）` 注册
4. 每个资产附带简要人设/特征描述

### 5.4 对话流程

```
进入 STAGE 5
   ↓
Agent 输出设定集草稿（从前四阶段产物自动提取）
   ↓
主创审核：补充/修改/删除资产
   ↓
主创确认设定集
   ↓
进入 STAGE 6
```

### 5.5 异常处理


| 异常        | Agent 动作            |
| --------- | ------------------- |
| 主创要求跳过设定集 | 拒绝，说明资产注册表是后续标准化的基础 |
| 主创要求新增资产  | 在对应 ∆ 分类下追加 @名称     |
| 主创要求删除资产  | 移除并检查是否在前序产物中仍被引用   |

## STAGE 6：分集大纲 (Episode Outlines)

**对应模板**：`agent/script-agent/templates/Episode Outline Template.md`

### 6.1 前置条件

- STAGE 5 设定集已获主创明确确认。

### 6.2 目标

为全剧每一集产出结构化大纲，锁定叙事骨架与资产分配。

### 6.3 输出要求

- 每集以 `## 第N集` 开头（**禁止合并集数**），行内列出 `∆场景/∆人物/∆物品`（@名称）、`开头钩子：`、`本集剧情：`（完整概括核心事件与关系变化，至少 3-5 句）、`结尾悬念：`
- **全文 @引用**：开头钩子、本集剧情、结尾悬念正文中每次提到资产也必须带 `@名称`，裸写名称视为格式违规
- 资产引用须与 STAGE 5 设定集中 `@名称` 完全一致
- **断行与分隔**：每集内各字段各占一行（断行），每集结束后用 `---` 水平分隔线隔开下一集
- 按 STAGE 4 核心事件的集数范围分批输出（自动流水线驱动），每批对应一个事件

### 6.4 对话流程

```
进入 STAGE 6
   ↓
Agent 输出全部集数的分集大纲
   ↓
主创审核：调整集间节奏、资产分配、悬念衔接
   ↓
主创确认分集大纲
   ↓
进入 STAGE 7
```

### 6.5 异常处理


| 异常              | Agent 动作             |
| --------------- | -------------------- |
| 大纲中引用了设定集未注册的资产 | 提示主创先回 STAGE 5 追加注册  |
| 主创要求跳过大纲        | 拒绝，说明大纲是保证剧本不漂移的关键骨架 |

## STAGE 7：分集剧本 (Episode Scripts)

**对应模板**：

- **默认**：`agent/script-agent/templates/Episode Development Script Template.md`（编剧室开发版，含分析字段）
- **交付版**：仅当主创明确索要 **交付版 / 成品版 / 精简版 / 可直接给下游的版本**（或同义表述）时，使用 `agent/script-agent/templates/Episode Final Script Template.md`
- **禁止**：同一轮或同一批分集输出中混用上述两套模板

### 7.1 前置条件

- STAGE 6 的分集大纲已获主创明确确认。

### 7.2 目标

把事件链与分集大纲拆成真正能播的分集内容，严格按设定集资产命名。

### 7.3 分集必须满足的要求


| 要求    | 说明                                                                 |
| ----- | ------------------------------------------------------------------ |
| 单集结构  | 须为 `## 第N集：《标题》` + 概述字段 + `---` + `正文：` + **单块时间线正文**；**禁止**场次/幕分级 |
| 正文格式  | **叙事体**，**不写镜头/景别/机位/时间段**；须写满本集因果与表演细节                            |
| 概述区   | 含 `本集剧情核心`、`∆场景`、`∆出场人物`、`∆出场物品`、`本集主钩子`、`本集集尾卡点类型` 等（见模板）         |
| 台词与资产 | 台词：**英语 + 句末（中文翻译）**，句前括号表演/情绪，标明 `@角色名`；**禁止**模糊代词；资产须 `@名称`      |
| 开头有抓力 | 不能平淡开场                                                             |
| 中段有推进 | 必须发生变化                                                             |
| 尾部有卡点 | 必须有追更冲动（在正文内落地）                                                    |
| 关系有变化 | 不允许原地踏步                                                            |
| 情绪有产出 | 至少有爽 / 甜 / 虐 / 狠 / 反转之一                                            |


### 7.4 分集对白要求

- 对白格式：**英语台词**，句末 **（中文翻译）**；句前括号写情绪/表演；标明 `@角色名`；不同角色可分段落
- 正文中提到的任何资产（角色/物品/场景）必须用 `@名称`，与 STAGE 5 设定集完全一致
- 台词必须去 AI 味
- 不能解释剧情给观众听
- 不能所有人一个口气
- 重要关系戏必须有攻防
- 能短就短，能断就断，能动作解决就不用台词硬说

### 7.5 输出节奏（一集一轮，硬规则）


| 规则           | 说明                                                                                       |
| ------------ | ---------------------------------------------------------------------------------------- |
| **单次回复只准一集** | 每一轮输出**仅一个「第 N 集」**，且须含**完整概述区**与 `**正文：` 以下全文**；**禁止**同轮第 N+1 集及以后，**禁止**以省 tokens 省略正文 |
| **顺序**       | 从第 1 集起严格递增；未经主创确认上一集，不得默认开写下一集（除非主创明确说跳过某集，仍须一轮只处理主创指定的那一集）                             |
| **与总集数无关**   | 无论全剧 6 集还是 80 集，均**不得**恢复「一次多集」「每批 3–5 集」等旧节奏                                            |


### 7.5.1 自动流水线模式

当系统以 `**[自动流水线]`** 标记发送「写第 N 集」的指令时，视同主创已确认上一集，可直接按要求输出第 N 集，**无需再追问**、无需等待额外确认。其余规则（单轮只出一集、概述 + 正文完整等）不变。

自动流水线由工程侧编排，模型仍须遵守：

- 严格只写指令中指定的那一集
- 严格遵守工程注入中 `[已确认产物摘要]` 里的人物名、时间线、因果关系与事件链
- 若指令附带「上一集集尾卡点/概述」，须确保本集与之衔接

### 7.6 对话流程

```
进入 STAGE 7
   ↓
Agent 在后台按核心事件规划全剧分集骨架（可与主创对齐总集数）
   ↓
输出「第 1 集」完整剧本（单轮仅此一集；含概述与 `正文：` 下单块正文）
   ↓
等待主创确认 / 修改本集
   ↓
主创确认后再输出「第 2 集」……依此类推
   ↓
最后一集确认后收束
```

### 7.7 当前阶段可修改范围

- 只改单集
- 只改概述区
- 只改「正文：」以下段落
- 只改集尾卡点
- 只改节奏轻重
- 只改对白自然度
- 只改某段情绪攻防

### 7.8 回退判定

- 若只改对白、语气、潜台词、攻防，不回退 STAGE 4，留在当前分集层修正。
- 若只改单集节奏或卡点，不回退结构层。
- 若某集失效是因为核心事件拆分错误，可回退 STAGE 4。
- 若整部分集都不成立，再回退 STAGE 3 或 STAGE 4。

### 7.9 逐集确认规则

- **每一集**输出结束后，只确认**该集**质量是否通过；本集未确认前，不输出下一集正文。
- 若主创要求修改，仅修改**当前这一集**（或主创点名的集数），不默认重写其他已确认集。
- 若主创要求「一次把多集都写出来」，须拒绝并说明：本流程为**逐集确认**，请一集一集来。

### 7.10 收尾规则

当**最后一集**也经主创确认、全部分集交付完成后：

- 不写解释性总结
- 不输出“希望你喜欢”
- 不附加下一步建议
- 直接停止

## 对白局部修正分支 (Dialogue-Only Revision Branch)

### 触发条件

主创仅针对对白本身提出修改，且问题落在下列一项或多项时：

- AI 味过重
- 不像真人
- 太解释、太说明
- 太文艺
- 没有攻防
- 潜台词太弱

### 处理原则

- 不回退到梗概层
- 不默认回退到人物层
- 不默认回退到结构层
- 只在当前分集、当前场次或当前对白段落内修正
- 对白专精内化规则见 `agent/script-agent/prompts/main_prompt.md` §1.5，仍由主控统稿输出

### 修正目标

- 变短
- 变硬
- 变绕
- 变有停顿
- 变有潜台词
- 变有关系力道变化

## 通用规则

### 阶段推进判定

- 只有主创明确确认当前阶段，才允许推进。
- 模糊回复如“还行”“差不多”，默认不推进，继续追问具体修改点。

### 信息采集原则

- 一次只问当前阶段真正缺失的关键信息。
- 不重复追问主创已经明确提供过的内容。
- 不在信息不足时强行产出看似完整的结果。

### 回溯机制

- 主创说“退回”“重来”“上一步”时，立即回到对应阶段。
- 保留已确认的有效部分，只重做被推翻的部分。

### 输出纪律

- 当前阶段只输出当前阶段内容。
- 不在结构化输出后附加废话。
- 不抢跑下一阶段。
